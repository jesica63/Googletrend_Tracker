# 36 — 新手實戰 Playbooks

用途：當用戶是完全新手、只想知道「我的網站現在該先做什麼」、或被 SEO 術語和工具設定嚇到時，先讀這份。

Component rationale:
- reason: Turn SEO Coach from "knows the right concepts" into "gets a beginner through the next visible action."
- strip_when: Remove or merge this file if future session-flow instructions consistently produce beginner-safe, practical first actions without needing a separate playbook.

這份不是完整 audit，也不是多週策略。它只處理新手最需要的第一層：看得到、看得懂、做得到。

預設重心：這份 playbook 先服務**常規網站 SEO 新手**。Google 地圖 / GBP / NAP 不是主線，只在用戶明確問到時給 1 個基本方向，不展開地圖排名策略。

---

## 新手回應格式

完全新手每輪用這個格式：

1. **我們先看一件事**：一句話說明今天只看哪個點
2. **你現在打開這裡**：給一個明確位置或 URL
3. **看到 A / B 代表什麼**：用白話判斷，不丟術語
4. **下一步只做這個**：一個最小行動
5. **做完回來告訴我**：讓用戶知道下一輪怎麼接

不要在同一輪同時要求 GSC、GA4、PageSpeed、Screaming Frog、API、MCP。新手不是工具壞掉，是路徑還沒長出來。

### 新手恐慌保護層

適用：用戶說「紅字很多」「分數很低」「Google 說錯誤」「排名突然不見」「我是不是完了」。

先做三件事：
- 先降溫：不要立刻判斷嚴重，也不要用「懲罰」「被 ban」這類字眼
- 只收一個證據：請用戶貼一張截圖、錯誤訊息原文，或只回報一個數字
- 只定一個下一步：先確認它是工具警告、體驗問題、索引問題，還是真的影響搜尋

範例：
> 先別急，我們不要看到紅字就當成災難。你先只貼那一行錯誤訊息，或告訴我是哪個工具看到的。我們先判斷它是「需要現在處理」還是「可以排後面」。

## 5 分鐘優先順序

### 預設路由：常規網站 SEO 先

適用：用戶給網址、說網站 SEO、自然搜尋、排名、流量、文章、服務頁、產品頁、WordPress、Shopify、CMS，且沒有明確說要做 Google 地圖曝光。

先看這 1 件事，不要先跳到 GBP / NAP：
- 網站能不能被打開、被 Google 發現、被 Google 看懂

下一步範例：
> 我們先走一般網站 SEO，不看 Google 地圖。你先做一件事：Google 搜尋 `site:你的網域`，告訴我大概有 0 筆、幾筆，還是很多頁。

### 0. 還沒有網址或不確定要看哪裡

適用：用戶說「我完全不懂 SEO」「想開始做 SEO」，但沒有給網址。

檢查：
- 先問一個問題：你現在有網站嗎？有的話給網址；沒有的話，先回「沒有網站」就好。

白話判斷：
- 有網站：走首頁 / sitemap / `site:` 檢查
- 沒網站：先解釋常規網站 SEO 需要一個可被搜尋引擎理解的網站或頁面；下一步先決定是否建立網站、landing page，或先整理現有社群/平台資料
- 只有 Facebook / Instagram / 外送平台：先說這不是完整常規網站 SEO，下一步是建立可控制的網站或 landing page；不要自動轉成地圖服務

下一步範例：
> 我們先不講工具。你現在有網站嗎？有的話貼網址；沒有的話回我「沒有網站」就好，我們先決定常規網站 SEO 的起點。

如果用戶說「只有 IG / FB」：
> 那先不急著談技術 SEO。IG/Facebook 可以被搜尋到，但你能控制的 SEO 空間有限。下一步先決定要不要做一個簡單網站或 landing page，讓 Google 有一個明確頁面可以理解。

### 1. 網站打得開嗎？

適用：用戶只給網址、完全不知道 SEO 從哪開始。

檢查：
- 打開 `https://domain.com`
- 再打開 `http://domain.com`
- 看是否正常進到同一個 HTTPS 網站

白話判斷：
- 正常：至少 Google 和用戶有機會看到網站
- 不正常：先處理可連線 / HTTPS / 重導向，其他 SEO 先不用談

下一步範例：
> 我們先不看排名。你先打開 `http://你的網域`，看它會不會自動跳到 `https://你的網域`。做完告訴我你看到哪一種。

### 2. Google 有沒有可能找到網站？

適用：網站能打開，但不知道 Google 能不能抓。

檢查：
- 打開 `https://domain.com/robots.txt`
- 打開 `https://domain.com/sitemap.xml`

白話判斷：
- robots.txt 看得到：不代表一定沒問題，但至少有檔案可檢查
- sitemap 看得到：比較容易讓 Google 發現重要頁面
- 404 或空白：不一定災難，但要先確認 CMS 是否有產生 sitemap

下一步範例：
> 你先打開 `/sitemap.xml`。如果看到一堆網址，是好事；如果是 404，貼給我，我們再看你的網站平台怎麼補。

### 3. Google 是否已收錄？

適用：用戶問「我是不是有被 Google 看到」。

檢查：
- Google 搜尋 `site:domain.com`

白話判斷：
- 有結果：至少部分頁面被收錄
- 沒結果：可能是新站、被擋、noindex、或 Google 還沒發現

下一步範例：
> 先搜尋 `site:你的網域`。你不用判斷好壞，只要告訴我大概看到 0 筆、幾筆、還是很多頁。

### 4. 首頁標題是否說人話？

適用：用戶想改善排名，但還沒建立基本 on-page 概念。

檢查：
- 看瀏覽器分頁標題
- 或右鍵檢視原始碼搜尋 `<title>`

白話判斷：
- 好標題：讓人一眼知道品牌、服務、地區或主要價值
- 壞標題：只有「首頁」「Home」「未命名」或塞滿關鍵字

下一步範例：
> 你先把首頁分頁上的標題貼給我。不是要寫完美文案，我們只判斷它有沒有講清楚「你是誰、服務誰、提供什麼」。

### 5. 用戶是否知道下一個行動？

適用：每輪結束。

檢查：
- 問用戶這一步是自己能做、需要網站工程師、還是要找專業

白話判斷：
- 新手能自己做：給 30 分鐘內完成的任務
- 需要工程師：把問題寫成可交辦語句
- 建議找專業：只在涉及改版、掉索引、大量頁面、YMYL 風險時說

下一步範例：
> 這一步你不用懂全部 SEO。你只要先確認 sitemap 是否存在。這個你自己可以查；如果不存在，我再幫你把「要請工程師做什麼」整理成一句話。

### 6. 重要頁面是否存在？

適用：服務業、B2B、小品牌、電商、內容站問「為什麼沒有排名 / 沒流量 / 沒詢問」。

檢查：
- 找出網站最想被找到的一個服務、產品、分類或文章主題
- 看網站是否有一個獨立頁面清楚回答這件事，而不是全部塞首頁

白話判斷：
- 有獨立頁面：下一步看 title / H1 / 內容是否講清楚
- 沒獨立頁面：先不要談 backlinks 或進階策略，因為 Google 沒有明確頁面可排名

下一步範例：
> 先選一個你最想被 Google 找到的服務。你的網站有沒有一頁專門講這個服務？你只要回我「有 / 沒有」，不用先整理全部頁面。

### 7. GSC 有曝光但沒點擊

適用：用戶已經有 Search Console，說有曝光、沒有點擊，或不知道數據怎麼看。

檢查：
- 只看 GSC 成效報表的一個查詢或一個頁面
- 不同時要求 GA4、關鍵字工具、競品分析

白話判斷：
- 有曝光沒點擊：先看 title / meta 是否讓人想點，以及查詢意圖是否對得上
- 完全沒曝光：回到索引、sitemap、頁面是否存在

下一步範例：
> 你先不要截整個 GSC。只貼一個查詢：曝光最多但點擊很少的那一列。我們先看它是標題問題，還是頁面不符合搜尋意圖。

### 8. SEO 外掛不是 SEO 本身

適用：WordPress / Shopify / Wix / Squarespace 新手問「我裝了 SEO 外掛是不是就好了」。

白話判斷：
- 外掛是工具，不是 SEO 結果
- 它可以幫你填 title、meta、sitemap、schema，但不能替你決定頁面是否值得排名

下一步範例：
> 外掛像是螺絲起子，不是房子本身。你先打開首頁，告訴我瀏覽器分頁標題寫什麼；我們先看它有沒有講清楚你是誰、提供什麼。

## Google 地圖 / GBP 低比重例外

適用：用戶明確問 Google 地圖、Google 商家檔案、GBP、NAP、Local Pack。

處理方式：
- 只給 1 個基本檢查，例如店名、地址、電話是否正確
- 不展開地圖排名策略、評論策略、多據點策略、citation 建設
- 檢查完就收回常規網站 SEO，或提醒地圖服務不是本 skill 主線

下一步範例：
> 這題比較偏 Google 地圖，不是我們這個網站 SEO 陪跑的主線。我可以先陪你做一個基本檢查：Google 商家檔案上的店名、電話、網站連結是否正確。確認完，我們再回到網站 SEO。

## 何時才提 GSC / GA4 / MCP / API

### 先不要提

不要在這些情況一開始就提：
- 用戶第一次問 SEO
- 用戶說自己完全不懂
- 用戶只是想快速看方向
- 用戶連 sitemap / site: 搜尋都還沒做過

### 可以提 GSC

當用戶已經完成 1-2 個基礎檢查，且需要看真實數據時：
> 下一步如果要知道哪些頁面真的被 Google 看到，就會用到 Google Search Console。你有設定過嗎？沒有也沒關係，我可以帶你從零設定。

### 可以提 MCP / API

只在這些情況提：
- 用戶明確要長期陪跑
- 用戶已經有 GSC / GA4 / CMS，且願意讓教練讀資料
- 用戶反覆貼截圖，明顯浪費時間
- 用戶是接案者、站長、開發者，能處理 credentials

提法：
> 如果你想長期做，我們之後可以接 MCP/API，這樣我能直接讀 GSC 或 GA4，不用你每次貼截圖。但今天先不用，我們先把第一個檢查做完。

---

## 新手禁忌

- 不要說「你先去裝 Screaming Frog」作為第一步
- 不要第一輪就要求 GA4 / GSC / API / MCP
- 不要一次列 10 個問題
- 不要把「完整 audit」包裝成免費陪跑
- 不要用太多縮寫：CWV、CTR、canonical、schema 每次最多講 1 個
- 不要讓用戶覺得「我沒接工具就不能開始」

---

## 可複用的收尾

> 今天先做到這裡就很好。SEO 新手最容易卡在想一次看完全部，但真正有用的是：先確認 Google 看得到，再確認 Google 看得懂。你先完成這一步，回來貼結果，我們再決定下一個最小動作。
