# 常見 SEO 情境（SEO Scenarios）

## 排名下降分析（Ranking Drop Analysis）

先給初學者一句話：
**先確認掉的是什麼，再猜為什麼掉。** 不要一看到流量下降就立刻大改網站。

### 第一步：確認問題的範圍
- 是所有頁面都下降，還是特定頁面？
- 是特定關鍵字下降，還是整體流量？
- 是自然搜尋流量下降，還是包含其他來源？

**工具：** GSC → 成效 → 比較日期範圍（下降前 vs 下降後）

### 常見原因排查

**1. Google 演算法更新**
- 確認下降時間點是否對應 Google 官方更新（搜尋 "Google algorithm update [月份]"）
- 如果對應 Core Update → 通常是 E-E-A-T 或內容品質問題
- 如果對應 Helpful Content Update → 審查是否有「為搜尋引擎而非人類」寫的內容

Coach 要補的預期管理：
- Core Update 不是「處罰」
- 很多時候不是你突然做錯，而是 Google 重新比較後，覺得別的內容更值得排前面
- 在 update 還沒完全跑完前，不要急著全部砍掉重寫

**2. 技術問題**
- 網站有沒有突然無法存取？
- robots.txt 是否意外封鎖了重要頁面？
- HTTPS 憑證是否過期？
- 有沒有大規模的 404 錯誤？

**3. 內容被刪除或更改**
- 是否刪除了頁面？
- 是否更改了 URL 但沒有設重定向？
- 是否把 noindex 加到重要頁面？

**4. 競爭對手變強**
- 搜尋你的目標關鍵字，看是否有新的強競爭者出現

**5. 失去重要反向連結**
- Ahrefs Webmaster Tools → 查看最近丟失的連結

## 網站改版的 SEO（SEO for Web Redesigns）

### 改版前必做清單
1. **抓取現有網站** 用 Screaming Frog 存下所有 URL 清單
2. **記錄排名** 在 GSC 截圖或匯出目前的關鍵字排名
3. **記錄流量基準** 匯出目前的月流量數據
4. **識別重要頁面** 哪些頁面帶來最多流量？這些必須被保護

### 改版中必做
1. **保持 URL 結構**（盡量不改變），如果必須改 → 一定要設 301 重定向
2. **維持內容**（特別是帶來流量的頁面內容）
3. **保持所有 meta tag**
4. **避免在改版同時換主機**（兩個變數同時改，難以排查問題）

### 改版後必做
1. 用 Screaming Frog 重新爬取，確認重定向設定正確
2. GSC 提交新 sitemap
3. 監控 GSC 的爬取錯誤
4. 追蹤 1-4 週的排名和流量變化

## 新網站上線（SEO for New Web Launch）

### 上線前確認清單
- [ ] HTTPS 已設定
- [ ] robots.txt 已設定（且沒有封鎖 Google）
- [ ] sitemap.xml 已生成
- [ ] 每個頁面有獨立的 title 和 meta description
- [ ] H1 標籤在每頁設定
- [ ] 所有圖片有 alt text
- [ ] 內部連結結構正確
- [ ] 結構化資料設定（至少首頁的 Organization schema）
- [ ] GSC 已驗證網站
- [ ] sitemap 已提交到 GSC

### 上線後的 SEO 期待
新網站通常需要 3-6 個月才能在競爭關鍵字上看到排名，這是正常的（俗稱「Google Sandbox」期）。

## Google Core Update 後的恢復

### Core Update 的特點
- 每年 3-4 次，通常持續 2-3 週才完全推出
- 主要評估整個網站的內容品質，而非技術問題
- 受影響後，通常要等下一次 Core Update 才能看到恢復

初學者白話版本：
- Core Update 比較像 Google 重新洗牌
- 不是說你犯規，而是它重新評估「誰更值得排前面」
- 所以恢復通常不是修一個小設定就立刻回來

### 如果你的網站被 Core Update 影響
1. **不要恐慌，不要立刻大規模修改**（讓 Update 完全推出後再評估）
2. 閱讀 Google 官方的「Self-assessment questions」（搜尋 "Google core update self-assessment"）
3. 對照問題逐一審查你的網站
4. 重點關注：E-E-A-T、內容品質、是否為人類寫作

可以這樣引導用戶：
- 先看是不是少數頁面掉，還是整站掉
- 再看掉的是不是原本就偏薄、偏舊、偏不明確的內容
- 最後才決定是補內容、補信任、還是先觀察

### 常見需要改善的方向
- 加強作者資訊和 E-E-A-T 信號
- 移除或改善低品質頁面
- 確保內容真正對用戶有幫助，而非只為了排名

---

### 不要做的事
- 不要 update 一來就一次改 100 個頁面
- 不要只因為焦慮就換 title、改 URL、砍大量內容
- 不要把所有問題都怪在演算法，先查技術和資料

## Coach 問法模板

當用戶說「我流量掉了」時，優先順序：

1. 「先別急，我們先看掉的是哪些頁、哪些關鍵字。」
2. 「這個時間點有沒有剛好遇到 Google 更新或網站改版？」
3. 「你最近有沒有改 URL、內容、robots、canonical 或主機設定？」

這樣比較像陪跑，也能避免一開口就亂猜。

## 網站遷移 SEO（Site Migration）

### 遷移類型
1. 換域名（example.com → newexample.com）
2. HTTP → HTTPS
3. 換主機/伺服器
4. 換 CMS（WordPress → Shopify 等）

### 最重要的原則：一次只做一件事
同時換域名又換 CMS，出問題時很難找原因。

### 遷移 SEO 標準流程
1. 準備：備份所有數據，記錄所有 URL
2. 設定：301 重定向所有舊 URL 到對應的新 URL
3. 更新：更新 sitemap、GSC 中的域名設定
4. 驗證：抓取確認重定向正確
5. 監控：持續追蹤 4-8 週

### 邊界聲明
複雜的網站遷移（大型電商、多語系、換域名）強烈建議找有遷移經驗的 SEO 專家協助，錯誤的遷移可能造成大量流量流失。
